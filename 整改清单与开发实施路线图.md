# `siyuan-copilot-opencode` 整改清单与开发实施路线图

## 1. 文档信息

- 文档目标：为 `siyuan-copilot-opencode` 制定一份完整的整改方案，解决与 `siyuan-copilot` 无法真正区分、共存时互相影响、安装顺序导致功能失效的问题。
- 适用对象：插件维护者、功能开发者、测试人员、发布人员。
- 编写日期：2026-03-23
- 审核基线：当前仓库代码状态
- 核心诉求：让本插件成为一个“可独立安装、可独立运行、可独立升级、可安全导入旧数据”的独立插件，而不是原插件的半共享 fork。

## 2. 背景与问题结论

### 2.1 当前结论

当前仓库虽然在 `plugin.json` 和 `package.json` 中使用了新的插件名 `siyuan-copilot-opencode`，但在以下层面仍然和 `siyuan-copilot` 发生耦合：

- 数据目录兼容逻辑长期保留，运行时会回退读取 `siyuan-plugin-copilot` 目录。
- 启动时会无条件导入旧插件数据。
- 设置合并使用浅合并，导致 fork 新增的 `opencode` 默认配置会被旧插件配置覆盖。
- 旧插件的迁移标记被一并复制，导致本插件误以为自己的会话迁移已经完成。
- 删除会话时会尝试删除旧插件目录中的数据。
- WebView 分区、全局事件、全局注入标记、图标 ID、tab type 等运行时命名空间仍然沿用 `copilot`。
- UI 标题、插件文案、远端 session title 也仍然保留原插件身份。

### 2.2 直接后果

- 如果用户先安装 `siyuan-copilot`，再安装本插件，本插件会读入旧插件配置。
- 由于浅合并，`opencode` provider 的默认配置会丢失或结构被污染。
- 用户看起来是在使用新插件，实际上很多数据仍在读原插件目录。
- 当用户删除会话或清理数据时，可能影响原插件数据。
- 两个插件共存时，WebView 登录态、全局事件、UI 图标和运行时标记都存在串扰风险。
- 用户无法清晰判断当前是哪个插件在工作，排查问题也会非常困难。

### 2.3 根因归类

根因可以分成五类：

1. 身份命名未彻底切换
2. 数据存储未彻底隔离
3. 配置 schema 未重构
4. 迁移机制设计错误
5. 运行时命名空间未隔离

## 3. 整改目标

### 3.1 总体目标

将本插件整改为一个真正独立的插件，实现以下结果：

- 可以与 `siyuan-copilot` 同时安装、同时启用、同时升级。
- 安装顺序不影响插件可用性。
- 本插件运行时不依赖旧插件目录。
- 本插件不会读写旧插件的实时数据。
- 旧插件数据只允许通过“一次性导入”进入本插件。
- `opencode` provider 的配置和运行逻辑成为正式的一等能力，而不是兼容层上的附着物。

### 3.2 具体目标

- 插件身份独立
- 存储空间独立
- 设置结构独立
- 迁移流程独立
- WebView 运行时独立
- UI 文案独立
- 测试与发布流程可验证

### 3.3 非目标

本次整改不以“新增更多 provider 功能”为主目标，以下内容不作为本轮核心交付：

- 新增大规模功能特性
- 重做整个聊天 UI
- 大规模修改提示词体系
- 重构所有历史 provider 的接口行为

如果某项调整与“插件独立化”无直接关系，应尽量后置。

## 4. 目标状态定义

整改完成后，系统应满足以下目标状态。

### 4.1 安装与运行

- `siyuan-copilot-opencode` 拥有完全独立的插件标识。
- `siyuan-copilot` 与 `siyuan-copilot-opencode` 可以并存。
- 两者的 tab、dock、command、icon、event、webview partition 不冲突。

### 4.2 数据与迁移

- 本插件只读写自己的数据目录。
- 旧插件数据只允许通过显式迁移流程导入。
- 迁移完成后，运行期不再回退读取旧插件目录。
- 本插件永远不会删除旧插件目录中的文件。

### 4.3 配置与 provider

- `opencode` 作为独立 provider 拥有自己的 schema。
- 设置文件具备版本号和迁移版本号。
- 设置合并采用深合并与 schema 补全，而不是对象浅覆盖。
- 即使用户来自旧版本或受污染版本，也能自动修复缺失的 `opencode` 配置。

### 4.4 可维护性

- 迁移逻辑和运行逻辑分层。
- 兼容逻辑只存在于迁移模块，不出现在运行时读写模块中。
- 所有命名空间常量集中管理。
- 发布前可通过测试矩阵验证并存能力。

## 5. 设计原则

### 5.1 一次性导入，长期隔离

兼容不等于长期双读双写。旧插件数据只能一次性导入，导入后立即脱钩。

### 5.2 迁移层与运行层分离

迁移逻辑只负责“读取旧数据并转换”，运行层只负责“操作当前插件数据”。

### 5.3 Schema 驱动，而不是结构猜测

设置、provider、session 元数据都应有明确 schema，不能靠对象存在性和弱约定推断。

### 5.4 命名空间必须彻底独立

只要是全局可见对象，就必须拥有插件专属前缀，包括但不限于：

- tab type
- dock type
- event name
- webview partition
- injected flag
- SVG symbol id
- temp dir
- 远端 session title

### 5.5 优先保护用户数据

只要存在误删、误覆盖、误串读可能，就必须优先选择更保守的设计。

## 6. 全面整改清单

下面按照模块给出完整整改清单。

## 6.1 模块 A：插件身份与品牌命名重构

### 目标

让插件在产品层、UI 层、运行时层都成为一个完全独立的实体。

### 必改项

1. 确定插件正式对外名称
   - 不再使用“Copilot OpenCode”这类仍然绑定原插件品牌的命名。
   - 明确插件英文名、中文名、简称、仓库说明、发布说明中的统一称呼。

2. 更新插件元信息
   - 更新 `plugin.json` 中 `displayName`、`description`、`keywords`。
   - 更新 `package.json` 中 `description`。
   - 更新 README 顶部描述，明确这是独立插件，不是共享运行态的扩展补丁。

3. 重命名 UI 标题
   - dock 标题
   - AI tab 标题
   - 新窗口标题
   - 右键菜单提示文案
   - WebView 默认图标与标题文案

4. 重命名远端会话 title
   - OpenCode session 创建时不再使用 `copilot-session`。
   - 使用本插件专属标识，例如 `opencode-plugin-session` 或更正式的新品牌名。

### 输出物

- 一套统一的品牌命名规范
- 更新后的 `plugin.json`、`package.json`、README、界面文案

### 验收标准

- 用户在界面中不会把本插件误认成原插件
- 日志、远端 session、设置页、dock、tab 均显示新插件身份

---

## 6.2 模块 B：全局命名空间与运行时标识重构

### 目标

切断与 `siyuan-copilot` 的运行时全局冲突。

### 必改项

1. 重命名 tab / dock type
   - `AI_SIDEBAR_TYPE`
   - `AI_TAB_TYPE`
   - `WEBAPP_TAB_TYPE`
   - 原则：全部改为带插件前缀的唯一值

2. 重命名 SVG symbol id
   - `iconCopilot`
   - `iconCopilotWebApp`
   - `iconWebApp_${appId}`
   - 统一为本插件前缀，例如 `iconOpenCodePluginMain`

3. 重命名全局事件
   - `copilot-summarize-doc`
   - 所有自定义事件名统一加前缀

4. 重命名注入标记
   - `window.__siyuan_copilot_injected_v3`
   - 所有 injected flag 改成新的唯一 key

5. 重命名 WebView console 协议消息
   - `__SIYUAN_COPILOT_HOTKEY__`
   - `__SIYUAN_COPILOT_LINK__`
   - 改为本插件专属消息头

6. 重命名 WebView 分区
   - 用户 WebView 分区
   - 抓取用隐藏 WebView 分区
   - 避免和原插件共用 cookie、登录态和缓存

7. 重命名临时目录
   - 例如 `siyuan_copilot`
   - 改为本插件独立临时目录名

### 输出物

- 一份“全局命名空间清单”
- 所有运行时全局 key 的统一常量定义文件

### 验收标准

- 同时启用两个插件时，不出现 tab/dock 错绑
- 两个插件图标不互相覆盖
- WebView 分区登录态互不干扰
- 注入脚本不会因另一个插件已注入而跳过

---

## 6.3 模块 C：存储层重构

### 目标

将“当前插件存储”和“旧插件导入”彻底分离。

### 必改项

1. 重构 `pluginPaths` 模块
   - 当前插件路径函数只负责当前插件目录
   - 旧插件路径函数只供迁移器使用
   - 运行时删除 `getPluginPathCandidates()` 这类双向候选逻辑

2. 明确目录边界
   - settings
   - sessions
   - assets
   - webappIcon
   - translate
   - prompts
   - agent-tools-config
   - webview-history

3. 删除运行时回退读取
   - `getPluginFileBlob()` 不能在运行期自动回退到旧插件目录
   - 当前插件缺文件时，应报错或走修复流程，不应该偷偷去旧目录读

4. 删除运行时对旧目录的写操作
   - 包括删除、更新、覆盖、清理

### 输出物

- 新的存储访问模块
- 单独的 legacy import 模块

### 验收标准

- 运行中所有读写路径都落在当前插件目录
- 搜索代码时，旧目录只在迁移器中出现

---

## 6.4 模块 D：迁移机制重构

### 目标

把当前“启动时偷偷复制旧数据”的逻辑，改成可控、可验证、可恢复的导入机制。

### 必改项

1. 移除 `onload()` 中的无条件导入
   - 启动时不应默认执行 legacy copy

2. 新建导入器
   - `LegacyImporter` 或同等模块
   - 明确导入来源、导入目标、导入清单、导入状态、导入报告

3. 定义导入流程
   - 检测旧插件是否存在
   - 检测当前插件是否已经导入
   - 提示用户选择是否导入
   - 备份当前数据
   - 导入设置
   - 导入 session 元数据
   - 导入 session 内容文件
   - 导入 assets
   - 导入 webapp icons
   - 导入 prompts / translate history / tools config
   - 写入导入报告

4. 引入迁移清单文件
   - 记录导入时间
   - 记录导入来源插件
   - 记录导入版本
   - 记录导入文件数
   - 记录成功/失败项

5. 导入后路径重写
   - 历史消息中的 asset 路径重写到当前插件目录
   - 不允许继续保留 legacy 路径引用

6. 迁移必须幂等
   - 重复执行不会产生重复数据或覆盖错误

7. 迁移失败可回滚
   - 至少支持导入前备份
   - 导入失败后不应留下半脏状态

### 输出物

- 独立导入器
- 导入状态文件
- 导入报告
- 用户交互入口

### 验收标准

- 用户不安装旧插件时，启动流程完全不受影响
- 用户安装旧插件时，可显式导入一次
- 导入完成后即使删除旧插件，本插件仍可正常工作

---

## 6.5 模块 E：设置系统与 Schema 重构

### 目标

解决设置被浅合并覆盖、provider 结构不稳定、迁移标记污染的问题。

### 必改项

1. 定义设置版本字段
   - `settingsVersion`
   - `migrationVersion`

2. 用 schema 驱动默认值
   - 所有设置项显式定义默认值与类型
   - 不再依赖对象是否存在来隐式补全

3. 实现深合并
   - `defaultSettings` 与用户设置合并时必须深合并
   - `aiProviders` 不能被整个覆盖
   - `opencode` 必须始终能被补齐

4. 区分“初始化补全”和“版本迁移”
   - 初始化补全：缺字段则补默认值
   - 版本迁移：旧结构升级为新结构

5. 重命名迁移标记
   - 不再沿用旧插件的 `dataTransfer.sessionData`
   - 改为本插件专属迁移命名

6. 启动自修复
   - 检测 `opencode` 配置缺失时自动恢复
   - 检测 providerOrder 缺项时补齐
   - 检测污染字段时自动矫正

### 输出物

- Settings schema
- 迁移器
- 深合并工具
- 启动自修复逻辑

### 验收标准

- 来自旧插件的设置不会覆盖掉 `opencode`
- 受污染用户升级后可自动恢复最小可用状态

---

## 6.6 模块 F：Provider 体系重构

### 目标

让 `opencode` 从“勉强塞进通用 provider 结构”变成正式的一等公民。

### 必改项

1. 单独定义 OpenCode provider schema
   - `serverUrl`
   - `models`
   - 如需扩展，可考虑 `workspacePolicy`、`sessionPolicy`、`toolPolicy`

2. 不再把 `opencode` 当通用 OpenAI 风格 provider 使用
   - 不应该强行补成 `{ apiKey, customApiUrl, models }`
   - `serverUrl` 应作为正式字段

3. 统一运行时读取入口
   - 业务层统一从 OpenCode config 读取 `serverUrl`
   - 老字段兼容只放在迁移层

4. 设置面板为 `opencode` 提供独立表单
   - 展示 server URL
   - 模型获取方式
   - 连接状态提示

5. provider 列表生成逻辑重构
   - 内置 provider 与自定义 provider 分开处理
   - 对特殊 provider 提供专用配置渲染

### 输出物

- OpenCode schema
- OpenCode provider adapter
- 独立设置表单

### 验收标准

- 无论是否导入旧插件设置，OpenCode 都可正常配置和使用
- 不再依赖 `customApiUrl` 作为 OpenCode 的隐式替代字段

---

## 6.7 模块 G：会话与资源管理重构

### 目标

确保会话、附件、图片、图标、翻译缓存全部在本插件目录中闭环。

### 必改项

1. 会话元数据与会话正文分层
   - 元数据文件
   - 独立 session 文件
   - 导入时一次性转移

2. 附件与图片路径重写
   - base64 转 asset 时只写新目录
   - 旧消息导入时把 legacy asset path 改写成新路径

3. 清除 legacy 路径正则兼容
   - 运行期不再支持同时解析两个插件目录
   - 统一解析当前插件目录

4. 图标缓存迁移
   - WebApp favicon
   - domain icon map
   - 清晰区分“导入图标”和“运行时缓存图标”

5. 删除只操作当前插件
   - 删除单个会话
   - 批量删除
   - 清理缓存
   - 卸载清理

### 输出物

- 干净的 session 读写模块
- asset path rewrite 逻辑
- 图标缓存管理器

### 验收标准

- 删除本插件会话不会影响旧插件
- 消息中不会残留旧插件目录路径

---

## 6.8 模块 H：WebView 与抓取逻辑重构

### 目标

隔离网页登录态、抓取态和页面注入逻辑，避免与原插件串扰。

### 必改项

1. 重新设计 WebView 分区
   - 用户主动打开的 WebApp 走本插件用户分区
   - 抓取网页内容的隐藏 WebView 可使用独立抓取分区
   - 禁止复用原插件分区名

2. 页面注入逻辑命名空间化
   - injected flag
   - console message 协议
   - dom-ready 注入逻辑

3. WebView 工具栏与搜索栏逻辑解耦
   - 不依赖原插件标识进行判断

4. WebApp 历史记录隔离
   - 历史记录文件只存本插件目录
   - 历史记录结构可加入版本字段

### 输出物

- 新的 WebView 常量定义
- 抓取分区策略说明

### 验收标准

- 两插件同时开网页登录页时不会共享登录状态，除非产品明确允许
- 抓取流程不会受另一插件注入标记影响

## 6.9 模块 I：UI、文档与用户认知整改

### 目标

让用户明确感知这是独立插件，并知道如何从旧插件安全迁移。

### 必改项

1. README 重写
   - 明确“支持与 `siyuan-copilot` 共存”
   - 明确“导入是一次性操作”
   - 明确“安装后不会共享运行期数据”

2. 设置页增加说明
   - 首次启动若发现旧插件，可展示导入建议
   - 如果检测到污染配置，展示修复提示

3. 新增迁移说明文档
   - 从旧插件迁移的步骤
   - 风险提示
   - 回退方案

4. 更新 CHANGELOG 发布说明模板
   - 明确这次是“独立化整改版本”

### 输出物

- 更新后的 README
- 用户迁移说明
- 发布说明模板

### 验收标准

- 普通用户能看懂如何从旧插件迁移
- 不会再误以为两个插件共享状态是正常设计

## 6.10 模块 J：自修复与故障恢复

### 目标

让已经被旧版本污染的用户在升级后能自动恢复。

### 必改项

1. 启动时检测异常状态
   - 当前插件 settings 中缺少 `opencode`
   - `migrationVersion` 缺失但 legacy 标记存在
   - session 元数据存在但正文文件缺失
   - providerOrder 丢失
   - `opencode` 被错误生成为通用 provider 结构

2. 自动修复流程
   - 补齐缺失 schema
   - 重置错误迁移标记
   - 必要时重新执行导入建议

3. 生成诊断报告
   - 便于用户反馈问题

### 输出物

- startup repair 模块
- repair report

### 验收标准

- 老用户升级后无需手工删配置即可恢复最小可用状态

---

## 6.11 模块 K：测试体系与发布流程重构

### 目标

让这次整改可验证、可回归、可发布。

### 必改项

1. 建立最小自动化测试基础
   - 设置迁移单测
   - 深合并单测
   - legacy import 单测
   - session path rewrite 单测

2. 建立手动回归清单
   - 只装本插件
   - 只装原插件
   - 先装原插件后装本插件

## 9. 验收标准

整改完成后，必须至少满足以下验收项。

### 9.1 安装与并存

- 只安装本插件，可以正常启动和使用
- 只安装原插件，不受本插件代码影响
- 先安装原插件，再安装本插件，本插件可正常使用
- 先安装本插件，再安装原插件，本插件仍可正常使用
- 两插件同时启用时，dock、tab、图标、WebView、事件不串扰

### 9.2 数据与迁移

- 导入旧插件数据后，本插件在旧插件卸载后仍可工作
- 删除本插件会话不会删除旧插件会话
- 本插件消息内不再引用旧插件 assets 路径

### 9.3 Provider 与设置

- `opencode` provider 在所有升级场景下均存在
- OpenCode 配置不会被旧插件设置覆盖
- 设置文件包含版本号与迁移标记

### 9.4 WebView

- WebView 登录态不与原插件意外共享
- 注入脚本不会被原插件标记阻断

### 9.5 用户体验

- 设置页能提示导入或修复
- README 清楚说明独立化与迁移方式

## 10. 风险清单

### 高风险

1. 去掉 fallback 后，历史污染用户可能一时看不到旧数据
2. 全局 type 改名后，旧布局缓存可能打不开
3. asset path rewrite 如果漏掉边缘格式，历史消息可能显示异常
4. WebView partition 切换后，用户登录态可能需要重新登录

### 中风险

1. 设置 schema 重构可能引入新兼容 bug
2. 发布说明不清晰会让用户误以为数据丢失

### 低风险

1. UI 文案改名带来的短期认知适应

## 11. 建议发布策略

### 11.1 版本策略

建议将本次整改视为一次“独立化版本”发布。

可选策略：

- 若插件品牌保持现有仓库名，仅做运行时彻底隔离，可使用一次较大的 minor 版本。
- 若插件品牌也发生明显变化，建议使用 major 版本，明确告诉用户这是独立化升级。

### 11.2 发布节奏

1. 内测版
2. 候选版
3. 正式版

### 11.3 发布说明重点

- 本版本解决与 `siyuan-copilot` 共存冲突问题
- 本版本将旧插件兼容逻辑改为一次性导入
- 升级前建议备份
- 升级后若检测到旧插件，可在设置中执行导入

## 12. 建议的落地顺序总结

如果只能按一个最稳妥的顺序实施，建议按下面顺序推进：

1. 先改全局命名空间
2. 再拆存储层与迁移层
3. 再重做设置 schema 和 OpenCode provider
4. 再处理 session / asset / icon 迁移
5. 再做 WebView 隔离
6. 最后补用户迁移、测试和发布

这样做的好处是：

- 先消除全局冲突，减少调试噪音
- 再消除数据串读，避免越修越乱
- 再保证配置正确，恢复核心功能
- 最后才处理用户升级体验和发布

## 13. Definition of Done

只有同时满足以下条件，才算本轮整改完成：

- 运行时不再读写 `siyuan-plugin-copilot` 目录
- 旧插件数据仅通过显式导入进入本插件
- 两插件可并存启用
- `opencode` provider 在所有升级路径下可用
- 删除、卸载、清理不会影响旧插件数据
- README 与设置页已明确说明迁移方式
- 测试矩阵已完成并记录结果

---

## 14. 结语

这次整改的本质不是“把几个字符串改一下”，而是把插件从“共享原插件运行时和数据的 fork”重构成“独立产品”。

只要还保留以下任何一种设计，本问题就会反复出现：

- 运行期双读旧目录
- 运行期双写旧目录
- 浅合并设置
- 共用 WebView partition
- 共用全局 injected flag
- 共用 tab/dock/event/icon 命名空间

因此，本整改必须按“身份、存储、配置、迁移、运行时”五条主线一起完成，不能只改其中一两处。
