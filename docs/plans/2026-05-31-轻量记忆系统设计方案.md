# 轻量记忆系统设计方案

> 日期：2026-05-31
> 状态：设计阶段，待实现
> 关联项目：siyuan-copilot-opencode（v0.0.2+）

---

## 1. 背景与动机

当前插件的 AI 交互已经能保存和恢复会话，但不同会话之间仍然隔离。Agent 可以看到当前窗口中的上下文，却不能自然地记住用户长期偏好、项目背景、反复出现的问题和上次对话的结论。

本方案要实现的是一个**轻量、透明、可编辑**的长期记忆系统：

- 记忆就是思源笔记中的 Markdown 文档
- 用户在设置页的“记忆”中选择一个“记忆笔记本”
- 插件只在这个笔记本内创建、读取、搜索和清理记忆
- 用户可以显式点击按钮，让 AI 扫描整个思源笔记仓库并生成 Agent 总览文档
- 不引入向量数据库、外部守护进程、嵌入模型或专用存储引擎

这个方向参考了若干开源记忆系统的共同经验：`agentmemory` 采用本地 Markdown 文件保存长期事实、日记和主题笔记；`axiomhq/agent-memory` 强调会后捕获、整合和分层披露；`opensite-skills` 将记忆拆成 episodic、semantic、procedural、working 等层；`Acontext` 主张 plain Markdown、可读可改、无 embeddings。这里不照搬它们的重检索能力，只借鉴“文件可读、结构稳定、渐进披露、会后提炼”的原则。

参考：

- https://github.com/jayzeng/agentmemory
- https://github.com/axiomhq/agent-memory
- https://github.com/opensite-ai/opensite-skills
- https://github.com/memodb-io/Acontext

## 2. 设计原则

1. **笔记本即边界**：用户选择一个思源笔记本作为记忆空间，所有记忆只写入和搜索这个笔记本。
2. **Markdown 即数据格式**：记忆文档必须能被用户直接阅读、编辑、移动、打标签、同步和导出。
3. **轻量优先**：第一阶段不做向量检索、不建 SQLite/图数据库、不引入后台服务。
4. **显式可控**：记忆功能默认可关闭；自动提取、注入数量、清除记忆都由用户控制。
5. **渐进披露**：核心档案少量常驻，情景记忆按关键词和时间检索后限量注入。
6. **幂等写入**：同一轮对话不能因为自动保存、切换会话或重试而重复生成多篇记忆。

## 3. 记忆笔记本布局

用户在设置页选择 `memory.notebookId`。插件在该笔记本内创建一个根目录，默认 `AI记忆`，也允许用户改为空字符串表示直接使用笔记本根目录。

```text
记忆笔记本/
  AI记忆/
    00-Agent总览.md
    01-核心档案.md
    02-情景记忆/
      2026/
        05/
          2026-05-27-1430-爬虫优化.md
    03-主题记忆/
      siyuan-copilot-opencode.md
      用户偏好.md
    90-反思草稿/
      2026-05-31-第10次对话后.md
    99-归档/
```

说明：

- `00-Agent总览.md` 是全库扫描后生成的总览文档，描述用户笔记仓库的主题、项目、知识结构和 Agent 工作建议。
- `01-核心档案.md` 是热记忆，记录稳定偏好、长期项目、沟通习惯。它可以复用现有 `soulDocId`。
- `02-情景记忆/` 是会话摘要，一轮或一次完整会话对应一篇文档。
- `03-主题记忆/` 是第二阶段的轻量语义记忆，按项目、人物、偏好、领域主题聚合。
- `90-反思草稿/` 只保存模型建议，默认不自动覆盖核心档案。
- `99-归档/` 保存压缩后的旧记忆，不默认删除用户笔记。

## 4. 记忆文档格式

### 4.1 情景记忆

每篇情景记忆是一篇普通思源文档。文档名包含时间和主题，文档内容使用 Markdown，并在正文开头放一段可读元信息。

```markdown
---
memory-id: ep_20260527_143012_ab12
memory-type: episodic
session-id: session_xxx
message-count: 8
message-hash: 8f3a1c
created-at: 2026-05-27T14:30:12+08:00
topics: Python, 爬虫, 性能优化
importance: 0.8
---

# 2026-05-27 爬虫优化

## 概述
用户讨论了一个 Python 爬虫性能问题，结论是优先改为 aiohttp + asyncio，并限制并发数。

## 可复用事实
- 用户正在做 Python 爬虫项目。
- 用户关心性能优化，但希望先保持实现简单。

## 后续线索
- 如果再次提到“昨天那个爬虫”，优先回忆本次讨论。

## 原始会话索引
- session-id: session_xxx
- 消息数: 8
```

约束：

- `memory-id` 稳定，不依赖标题，方便改名后仍能识别。
- `message-hash` 用于幂等，避免重复提取。
- 默认只保存摘要和关键事实，不保存完整原文；可以提供“保存完整对话”开关。

### 4.2 核心档案

核心档案是用户可直接编辑的固定文档。若用户已设置 `soulDocId`，优先把该文档视为核心档案，不再创建重复概念；若未设置，初始化时在记忆笔记本中创建 `01-核心档案.md` 并记录其文档 ID。

```markdown
# Agent 核心档案

## 用户偏好
- 喜欢直接、少废话、可执行的建议。
- 做技术方案时偏好轻量实现，不喜欢过早引入复杂基础设施。

## 当前长期项目
- siyuan-copilot-opencode：Svelte + TypeScript 的思源插件，连接本地 OpenCode 服务。

## 沟通约定
- 当任务涉及代码修改时，先核对现有架构，再给最小可落地方案。

## 成长记录
- 2026-05-31：确认记忆系统采用思源笔记本内 Markdown 存储，不使用向量数据库。
```

---

### 4.3 Agent 总览文档

Agent 总览文档由用户在设置页点击“扫描笔记仓库”后生成。它不是每轮对话自动更新的内容，而是一次显式的全库探索结果。

```markdown
# Agent 总览

## 仓库概况
- 笔记本数量：6
- 主要内容：项目开发、读书笔记、日常记录、知识管理、AI 工具实验。

## 主要主题
- siyuan-copilot-opencode：当前活跃开发项目。
- AI Agent 与工具调用：长期关注方向。
- 个人知识管理：围绕思源、双链、任务和复盘展开。

## 可能的长期偏好
- 偏好轻量、可控、本地优先的方案。
- 不喜欢过早引入复杂基础设施。

## 给 Agent 的工作建议
- 回答前优先考虑本地笔记中已有项目背景。
- 遇到跨项目问题时，先确认用户当前关注的项目。

## 扫描信息
- generated-at: 2026-05-31T10:00:00+08:00
- scanned-notebooks: 全部未关闭笔记本
- source: AI 自动扫描摘要，用户可编辑
```

生成规则：

- 输出写入记忆笔记本的 `{rootPath}/00-Agent总览.md`。
- 若文档已存在，默认创建“覆盖前预览”或写入一个新版本小节，不静默覆盖用户修改。
- 总览内容可作为系统提示中的高优先级背景，但仍以用户当前输入为准。

## 5. 设置页设计

现有设置页中间的 `SOUL 文档` 分组改名为 `记忆`。这个分组不再只配置 SOUL 文档 ID，而是成为轻量记忆系统的统一入口。

### 5.1 记忆设置分组

分组名称：

- 中文：`记忆`
- 英文：`Memory`

包含控件：

| 控件 | 类型 | 说明 |
|------|------|------|
| 启用长期记忆 | 开关 | 总开关，关闭后不检索、不注入、不自动提取 |
| 灵魂文档 ID | 文本输入 + 验证按钮 | 原 `soulDocId`，作为核心档案优先来源 |
| 记忆笔记本 | 下拉选择 | 选择记忆存储笔记本，保存为 `memory.notebookId` |
| 记忆根目录 | 文本输入 | 笔记本内相对路径，默认 `AI记忆` |
| 自动提取记忆 | 开关 | 回复完成后是否异步生成情景记忆 |
| 保存完整对话 | 开关 | 默认关闭，只保存摘要和事实 |
| 注入数量与长度 | 数字输入或滑块 | 控制情景记忆数量、主题记忆数量、总字符预算 |
| 扫描笔记仓库 | 按钮 | 显式扫描整个思源笔记仓库并生成 Agent 总览 |
| 查看 Agent 总览 | 按钮 | 打开 `{rootPath}/00-Agent总览.md` |
| 查看核心档案 | 按钮 | 打开 `soulDocId` 或 `memory.coreDocId` |
| 清除插件记忆 | 危险按钮 | 二次确认后删除记忆根目录下插件生成的内容 |

交互要求：

- 未选择记忆笔记本时，扫描、自动提取和查看记忆按钮禁用。
- `灵魂文档 ID` 保留现有验证能力，但 UI 文案从 `SOUL 文档 ID` 改为 `灵魂文档 ID`。
- `记忆笔记本` 应复用现有笔记本列表能力，展示笔记本名称，保存笔记本 ID。
- 扫描按钮点击后必须二次确认，因为它会读取大量笔记内容并调用 OpenCode 生成摘要。
- 扫描过程中显示进度、当前阶段和可取消按钮。

### 5.2 全库扫描流程

“扫描笔记仓库”是用户主动触发的重操作，和日常聊天检索分开。

流程：

1. 用户在“记忆”设置中选择记忆笔记本。
2. 用户点击“扫描笔记仓库”。
3. 插件弹出确认框，说明将读取整个思源笔记仓库的文档标题、路径和必要正文片段。
4. 插件列出所有未关闭笔记本，排除记忆根目录，避免把旧记忆反复总结进总览。
5. 先收集文档标题、路径、更新时间、标签和少量正文片段，生成仓库目录摘要。
6. 对较大的笔记本分批摘要，再把批次摘要交给 OpenCode 合并为 Agent 总览。
7. 将总览写入 `{memory.rootPath}/00-Agent总览.md`。
8. 保存 `memory.overviewDocId`、`pluginData.memoryOverviewScanState.lastScanAt`、扫描范围和文档数量。

边界：

- 不建立长期索引，不保存向量。
- 不把全库正文长期复制到插件私有文件。
- 默认只生成总览，不为每篇笔记生成记忆文档。
- 失败时保留已生成的批次摘要草稿，方便重试。

## 6. 检索与注入机制

### 6.1 检索范围

检索只在用户选择的记忆笔记本内进行：

- 必须满足 `box = settings.memory.notebookId`
- 如果配置了 `memory.rootPath`，还必须满足文档路径位于该根目录下
- 只搜索 `00-Agent总览.md`、`01-核心档案.md`、`02-情景记忆/`、`03-主题记忆/`，默认跳过 `90-反思草稿/` 和 `99-归档/`

### 6.2 检索策略

第一阶段不使用模型做“意图分析”，避免每轮都增加一次额外调用。采用确定性策略：

1. 始终读取核心档案和 Agent 总览，但按字符预算截断。
2. 从当前用户输入中提取关键词：中文词组、英文标识符、日期词、文件名、项目名、标签。
3. 先调用思源文档搜索能力获取候选文档，再按 `notebookId + rootPath` 过滤。
4. 搜索结果不足时，用 SQL 在 `blocks` 表中按 `box`、`path/hpath`、`content LIKE` 做补充查询。
5. 排序时综合：关键词命中数、文档更新时间、`importance`、是否主题记忆。
6. 默认注入：
   - Agent 总览片段：最多 3000 字符
   - 核心档案片段：最多 4000 字符
   - 情景记忆：最多 5 条，每条最多 800 字符
   - 主题记忆：最多 2 条，每条最多 1200 字符

### 6.3 Prompt 注入

记忆注入发生在 `ai-sidebar.svelte` 的 `prepareMessagesForAI()` 阶段，而不是 `ai-chat.ts`。

原因：

- 现有系统提示、SOUL 文档、上下文文档和消息截断都在 `prepareMessagesForAI()` 中统一处理。
- `ai-chat.ts` 只负责把最终 messages 扁平成 OpenCode prompt，不适合做检索和业务决策。

注入方式：

- 不新增特殊 role。
- 将记忆内容拼接到最终 system prompt 中。
- 如果 `soulDocId` 已经作为核心档案注入，则不要重复注入同一文档。

示例：

```text
=== 长期记忆 ===

Agent 总览：
- 用户笔记仓库主要包含思源插件开发、AI Agent、知识管理和日常复盘。
- 当前活跃项目是 siyuan-copilot-opencode。

核心档案：
- 用户偏好轻量方案，不希望引入向量数据库。

相关往事：
- [2026-05-27 爬虫优化] 用户希望优化 Python 爬虫性能，倾向 aiohttp + asyncio。

使用规则：
- 这些内容是可参考的长期记忆，不要声称绝对确定。
- 如果当前问题与记忆冲突，以用户当前输入为准。
```

## 7. 写入与幂等触发

### 7.1 写入时机

不要把“保存会话”当成“会话结束”。现有 `saveCurrentSession()` 会在自动保存、流式回复完成、多模型选择、切换会话等多个时机触发，直接挂在它后面会重复生成记忆。

第一阶段采用两个入口：

1. **自动提取**：一轮 assistant 回复完成并保存成功后，异步触发一次提取。
2. **手动记住**：用户在聊天界面对当前轮或当前会话点击“记住”。

### 7.2 幂等规则

提取前计算：

- `sessionId`
- 当前非 system 消息数
- 最后一条 assistant 消息内容 hash
- 最近一条 user 消息内容 hash

写入后在插件设置的 `pluginData.memoryExtractionState` 中记录：

```json
{
  "session_xxx": {
    "messageCount": 8,
    "messageHash": "8f3a1c",
    "memoryDocId": "20260527143012-abc123",
    "updatedAt": 1780000000000
  }
}
```

若同一 `sessionId + messageCount + messageHash` 已提取过，则跳过。

### 7.3 提取模型调用

提取阶段调用 OpenCode，但必须使用独立的内部 prompt，不污染当前对话。返回 JSON 后再由插件生成 Markdown。

返回结构建议：

```ts
interface ExtractedMemoryDraft {
  title: string;
  summary: string;
  facts: string[];
  followUps: string[];
  topics: string[];
  importance: number;
  shouldRemember: boolean;
}
```

若 `shouldRemember = false` 或重要性低于用户配置阈值，则不写入文档。

## 8. 设置项

新增配置块：

```ts
memory: {
  enabled: false,
  notebookId: '',
  rootPath: 'AI记忆',
  overviewDocId: '',
  autoExtract: true,
  saveFullConversation: false,
  maxOverviewChars: 3000,
  maxCoreChars: 4000,
  maxEpisodicItems: 5,
  maxTopicItems: 2,
  maxMemoryPromptChars: 12000,
  minImportance: 0.35,
  coreDocId: '',
}
```

说明：

- `notebookId` 是必选边界。未选择笔记本时，记忆功能不启用。
- `rootPath` 是笔记本内相对路径，不是全局路径。
- `overviewDocId` 是全库扫描生成的 Agent 总览文档 ID。
- `coreDocId` 是插件创建的核心档案文档 ID；如果 `soulDocId` 存在，则优先使用 `soulDocId`。
- 幂等状态、反思计数、总览扫描状态等运行数据放入 `pluginData`，不放入用户可调设置。

运行数据建议：

```ts
pluginData: {
  memoryExtractionState: Record<string, MemoryExtractionState>,
  memoryOverviewScanState: {
    lastScanAt: number,
    lastScanDocCount: number,
    lastScanNotebookIds: string[],
    status: 'idle' | 'running' | 'failed' | 'completed',
  },
}
```

## 9. 两阶段实现路线

### 第一阶段：轻量可用

目标：Agent 能跨会话读取 Agent 总览、核心档案和相关会话摘要。

新增模块：

- `src/memory/types.ts`：类型定义。
- `src/memory/storage.ts`：笔记本内目录初始化、文档创建、文档读取、路径过滤。
- `src/memory/retriever.ts`：基于思源搜索和 SQL fallback 的关键词检索。
- `src/memory/injector.ts`：把记忆上下文拼进 system prompt。
- `src/memory/extractor.ts`：会后提取摘要并写入情景记忆。
- `src/memory/scanner.ts`：显式扫描整个思源笔记仓库，生成 Agent 总览。
- `src/memory/sanitize.ts`：敏感信息过滤和 Markdown 清洗。

集成点：

| 集成点 | 修改内容 |
|--------|----------|
| `defaultSettings.ts` | 新增 `memory` 默认配置和 `pluginData.memoryExtractionState` |
| `settingsSchema.ts` | 深合并并规范化 `memory` 配置 |
| `SettingsPanel.svelte` | 将 `SOUL 文档` 分组改名为 `记忆`，加入记忆笔记本、根目录、扫描按钮 |
| `src/index.ts` / 插件 onload | 如果已启用且有笔记本，初始化记忆目录、Agent 总览和核心档案 |
| `ai-sidebar.svelte` / `prepareMessagesForAI()` | 检索并注入 Agent 总览、核心档案和相关记忆 |
| `ai-sidebar.svelte` / 回复完成后 | 异步触发记忆提取，使用幂等记录避免重复 |

第一阶段不做：

- 向量检索
- 图数据库
- 后台守护进程
- 自动改写核心档案
- 跨笔记本搜索
- 自动后台全库扫描

### 第二阶段：反思草稿和主题记忆

目标：让 Agent 能整理旧记忆，但仍保持用户可控。

功能：

- 每 N 次有效记忆后生成一篇反思草稿，写入 `90-反思草稿/`。
- 反思草稿列出建议新增、修改、删除的核心档案条目。
- 设置页提供“应用到核心档案”按钮，用户确认后才写入。
- 可将多个情景记忆合并为 `03-主题记忆/项目名.md`。
- 每月可生成月度摘要并把旧情景记忆移动到 `99-归档/`，默认不删除。

## 10. UI/UX 设计

### 10.1 设置页

将现有 `SOUL 文档` 设置分组改为 `记忆` 设置分组：

- 启用长期记忆
- 灵魂文档 ID
- 选择记忆笔记本

### 10.2 扫描按钮状态

扫描按钮至少包含四个状态：

- `未配置`：未选择记忆笔记本时禁用。
- `可扫描`：点击后弹确认框。
- `扫描中`：显示进度、当前笔记本或批次、取消按钮。
- `已完成`：显示上次扫描时间、文档数，并提供打开 Agent 总览。

扫描失败时显示错误摘要，并允许重试。重试应复用已收集的安全中间结果，不重复写入总览文档。

### 10.3 聊天界面

- 回复下方显示“使用了 X 条记忆”。
- 点击后展开本轮注入的记忆标题和来源文档链接。
- 用户可对当前轮点击“记住”，手动触发提取。
- 如果记忆功能未配置，界面不打扰用户。

## 11. 数据安全与隐私

- 所有记忆默认保存在用户选择的思源笔记本中，随思源自身同步策略走。
- 记忆提取会把当前会话摘要请求发送给本地 OpenCode 服务；用户可关闭自动提取。
- 全库扫描会读取用户所有未关闭笔记本的标题、路径、标签和必要正文片段，并发送给本地 OpenCode 服务生成分批摘要；必须由用户点击按钮并确认后才运行。
- 默认不保存完整对话，只保存摘要和可复用事实。
- 写入前用规则过滤明显敏感内容，例如 API Key、Token、密码、私钥片段。
- 清除记忆只删除插件在记忆根目录下创建的内容，并二次确认。
- 自动反思只生成草稿，不默认覆盖核心档案。

## 12. 风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| 关键词检索漏召回 | 中 | 核心档案常驻，情景记忆用标题、主题、正文共同检索；后续再考虑轻量索引 |
| 自动保存导致重复记忆 | 高 | 使用 `sessionId + messageCount + messageHash` 幂等记录 |
| 记忆注入过长 | 中 | 限制核心档案、单条记忆、总注入字符数 |
| 全库扫描耗时长 | 中 | 只在用户点击后运行，分批扫描，可取消，生成进度提示 |
| Agent 总览误概括 | 中 | 写入可编辑文档，不自动覆盖用户修改；扫描结果标明生成时间和来源 |
| 模型提取出错误事实 | 中 | 默认写摘要，不自动改核心档案；记忆文档用户可编辑 |
| 误存敏感信息 | 中 | 默认不保存完整对话，写入前规则过滤，提供手动删除 |
| 多设备同步冲突 | 低 | 依赖思源同步；文档 ID 不稳定时用 `memory-id` 和路径兜底 |

## 13. 与现有架构的兼容性

- 关闭记忆后，插件行为应与当前版本一致。
- 不修改 OpenCode provider 的职责，provider 仍只负责请求和流式响应。
- 记忆注入在 `prepareMessagesForAI()` 里完成，和现有系统提示、SOUL 文档、上下文文档保持同一入口。
- `soulDocId` 优先作为核心档案使用；没有 `soulDocId` 时才创建 `memory.coreDocId`。
- 文档创建继续复用 `createDocWithMd(notebook, path, markdown)`，因此设置中必须保存 `notebookId`。
- 搜索复用现有 `searchDocs` 和 `sql` API 包装，所有结果按记忆笔记本和根目录过滤。
- 全库扫描复用思源笔记本列表、文档列表、Markdown 导出和 OpenCode 调用能力；扫描结果仍写回记忆笔记本。

## 14. 成功标准

第一阶段完成后：

- [ ] 用户在设置页选择一个记忆笔记本后，插件能初始化记忆目录和核心档案。
- [ ] 设置页中原 `SOUL 文档` 分组改名为 `记忆`，且可以同时配置灵魂文档和记忆笔记本。
- [ ] 点击“扫描笔记仓库”后，插件能扫描整个思源笔记仓库并在记忆笔记本中生成 `00-Agent总览.md`。
- [ ] 一轮对话完成后，`02-情景记忆/` 下自动生成一篇非重复记忆文档。
- [ ] 第二天提到相关主题时，Agent 能引用前一天的摘要。
- [ ] 关闭记忆开关后，消息发送、会话保存和 SOUL 文档行为不变。
- [ ] 搜索和注入只发生在用户选择的记忆笔记本内。

第二阶段完成后：

- [ ] 插件能生成反思草稿，但不会未经确认改写核心档案。
- [ ] 用户可以把反思草稿应用到核心档案。
- [ ] 旧情景记忆可以被压缩为月度摘要或主题记忆，并移动到归档目录。

## 15. 后续实现计划提示

进入实现计划时，优先拆成以下任务：

1. 设置与初始化：将 `SOUL 文档` 分组改名为 `记忆`，加入 `memory.notebookId`、目录创建、核心档案创建或复用 `soulDocId`。
2. 全库扫描：实现设置页扫描按钮、分批读取、OpenCode 总结和 `00-Agent总览.md` 写入。
3. 检索与注入：在 `prepareMessagesForAI()` 中加入 Agent 总览、核心档案和情景记忆。
4. 会后提取：回复完成后异步生成情景记忆，并实现幂等。
5. UI 透明度：设置页、手动“记住”、本轮使用记忆提示。
6. 第二阶段再做反思草稿、主题记忆和归档。

---

*本设计保持轻量路线：日常记忆只存在于用户选择的思源笔记本中；全库扫描仅在用户显式点击后运行，用于生成可编辑的 Agent 总览文档，第一阶段不引入向量数据库或额外存储服务。*
