# 全局长期记忆 Agent 设计方案

> 日期：2026-05-28
> 状态：设计阶段，待实现
> 关联项目：siyuan-copilot-opencode（v0.0.2+）

---

## 1. 背景与动机

当前插件的 AI 交互模式是**无状态对话框**：每次打开侧边栏，Agent 对用户一无所知；每次对话，仅依赖当前窗口内的消息历史。虽然会话可以保存和恢复，但不同会话之间完全隔离，Agent 无法跨对话学习用户的偏好、习惯或积累项目知识。

用户期望的是一个**有长期记忆、能不断进化**的 Agent：它应该记得用户是谁、喜欢什么、在做什么项目，并且随着时间的推移越来越"懂"用户。

---

## 2. 设计目标

1. **全局记忆**：所有对话共享同一个记忆空间，Agent 在任何会话中都能访问历史积累的知识
2. **分层存储**：模仿人类记忆模型，区分短期工作记忆、中期情景记忆和长期核心档案
3. **自动进化**：Agent 能主动从对话中提取知识、总结规律、更新自我认知
4. **渐进实现**：分两阶段落地，第一阶段先让 Agent"记得住"，第二阶段再让 Agent"会思考"
5. **笔记即记忆**：所有记忆数据直接存储为思源笔记文档，用户可自定义路径，充分利用思源的搜索、标签、双向链接和同步能力

---

## 3. 核心概念：分层记忆模型

### 3.1 工作记忆（Working Memory）

- **作用**：当前对话的实时上下文
- **存储**：内存中的 `messages[]` 数组
- **生命周期**：单次会话，关闭即释放
- **现有状态**：✅ 已实现（`ai-sidebar.svelte` 中的消息列表）

### 3.2 情景记忆（Episodic Memory）

- **作用**：每次对话的"往事"摘要。Agent 可以"回想起"和用户之前的某次交流
- **存储**：每篇记忆存为**一篇思源笔记文档**，放在用户指定的路径下（如 `/AI记忆/情景记忆/2026/05/27-爬虫优化.md`）
- **文档格式（Markdown）**：
  ```markdown
  ---
  memory-id: ep_1716883200000
  session-id: session_xxx
  timestamp: 1716883200000
  mood: 积极
  topics: Python, 爬虫, 性能优化
  importance: 0.8
  ---

  # 与用户的对话摘要（2026-05-27）

  ## 概述
  用户让我帮他优化一个 Python 爬虫的性能问题，我建议他使用 aiohttp 和 asyncio。

  ## 关键事实
  - 用户正在写一个 Python 爬虫项目
  - 用户关心性能优化
  - 推荐技术栈：aiohttp + asyncio

  ## 完整对话
  [可选：折叠的完整对话记录，便于回溯]
  ```
- **写入时机**：每次对话结束后（或用户主动点击"记住这次对话"）
- **写入方式**：由 OpenCode 模型自动生成摘要，插件调用思源 API 创建文档
- **优势**：用户可直接在笔记中浏览、搜索、编辑、打标签；天然支持思源的双向链接和图谱

### 3.3 核心档案（Core Profile）

- **作用**：Agent 的"灵魂"和用户的"画像"。包含深层偏好、固定习惯、跨项目知识体系
- **存储**：一篇固定的**思源笔记文档**（如 `/AI记忆/核心档案.md`），用户可直接查看和编辑
- **文档格式（Markdown）**：
  ```markdown
  ---
  doc-type: agent-core-profile
  updated-at: 1716883200000
  ---

  # 我的 Agent 档案

  ## Agent 自我认知
  - **性格**：耐心、简洁、技术导向
  - **沟通风格**：优先给代码示例，再解释原理
  - **成长记录**：
    - 2026-05-20：发现用户对前端框架感兴趣，调整了技术推荐倾向

  ## 用户画像
  - **身份**：全栈开发者，偏好 Python + TypeScript
  - **习惯**：
    - 喜欢先写伪代码再实现
    - 不喜欢冗长的解释，偏好代码+简短注释
  - **当前项目**：
    - siyuan-copilot-opencode（TypeScript + Svelte + SiYuan Plugin，活跃开发中）
  - **技术图谱**：
    - 精通：Python, Django, React
    - 熟悉：Docker, Kubernetes
    - 学习中：Rust, WebAssembly

  ## 我们的关系
  - **累计对话**：42 次
  - **初次相识**：2026-05-01
  - **信任度**：高
  ```
- **更新时机**：主动反思阶段（见第5节）
- **优势**：用户随时可以在思源笔记中直接编辑 Agent 对自己的认知；支持 Markdown 的所有表达能力（表格、列表、引用等）

## 4. 记忆检索与注入机制

### 4.1 检索流程

当用户发送一条新消息时：

1. **意图分析**：先让模型快速判断用户是否需要历史记忆（例如"帮我改一下昨天那个脚本"需要记忆，"翻译这句话"不需要）
2. **思源全文检索**：调用思源 SQL API，在 `rootPath/情景记忆/` 路径下全文搜索用户当前输入的关键词
3. **时间排序**：优先返回最近的记忆，辅以相关度排序
4. **核心档案读取**：读取核心档案文档，提取与用户问题相关的段落（关键词匹配）
5. **排序与截断**：按综合相关性排序，取 Top-K（默认 5 条情景记忆 + 核心档案的对应部分）
6. **注入 prompt**：将检索结果格式化为"你的记忆"，以 system message 或特殊的 memory message 形式注入

### 4.2 Prompt 注入格式示例

```
[系统提示词]
...

[你的记忆]
关于用户：
- 用户是全栈开发者，偏好 Python + TypeScript
- 用户正在开发 SiYuan 插件（siyuan-copilot-opencode）
- 用户喜欢先写伪代码再实现，不喜欢冗长解释

相关往事：
- [2026-05-27] 用户让我帮他优化一个 Python 爬虫，我推荐了 aiohttp + asyncio
- [2026-05-26] 用户在配置 Vite 的 external 选项时遇到问题，我帮他排除了 siyuan 和 child_process

[用户当前输入]
...
```

## 5. 两阶段实现路线

### 第一阶段：基础记忆（让 Agent "记得住"）

目标：实现核心档案 + 情景记忆的存储和检索，Agent 能跨对话引用历史信息。

#### 5.1.1 数据存储层

- **记忆根路径**：用户可自定义（默认 `/AI记忆/`），存在设置中 `memory.rootPath`
- **核心档案**：`{rootPath}/核心档案.md`（思源文档，用户可编辑）
- **情景记忆**：`{rootPath}/情景记忆/YYYY/MM/DD-主题.md`（按日期分层，避免单目录过多）

**初始化流程**：
```
plugin.onload()
  → 检查 settings.memory.rootPath 是否设置
    → 未设置：弹出引导，让用户选择/创建记忆根目录（如 `/AI记忆/`）
    → 已设置：检查核心档案文档是否存在，不存在则自动创建模板
```

#### 5.1.2 记忆提取模块（`src/memory/extractor.ts`）

功能：
- `extractEpisodicMemory(sessionMessages: Message[]): EpisodicEntry`
  - 调用 OpenCode 模型，传入整个对话历史
  - 模型返回 JSON 格式的摘要、关键事实、主题、重要性评分
  - 插件调用思源 API（`createDocWithMd`）在 `rootPath/情景记忆/YYYY/MM/` 下创建新文档
- `updateCoreProfile(entry: EpisodicEntry): void`
  - 将新记忆与现有核心档案合并（简单追加 keyFacts，去重）
  - 第一阶段暂不启用，只做接口预留

#### 5.1.3 记忆检索模块（`src/memory/retriever.ts`）

功能：
- `retrieveRelevantMemories(query: string, limit: number): MemoryContext`
  - **检索情景记忆**：调用思源 SQL API，在 `rootPath/情景记忆/` 路径下全文搜索 `query`，按时间倒序取 Top-K
  - **检索核心档案**：读取核心档案文档，提取与 `query` 关键词匹配的段落
  - 返回排序后的记忆列表 + 核心档案的对应片段

#### 5.1.4 Prompt 注入模块（`src/memory/injector.ts`）

功能：
- `injectMemories(messages: Message[], memories: MemoryContext): Message[]`
  - 在消息数组的最前面插入一个 `role: 'system'` 的 memory message
  - 或者替换现有的 system prompt，附加记忆段落

#### 5.1.5 与现有系统的集成点

| 集成点 | 修改内容 |
|--------|----------|
| `ai-chat.ts` 的 `chat()` 调用前 | 插入 `retriever.retrieveRelevantMemories(userInput)` 和 `injector.injectMemories()` |
| `ai-sidebar.svelte` 的会话结束 | 在 `saveCurrentSession()` 之后调用 `extractor.extractEpisodicMemory()` |
| `settingsSchema.ts` | 新增 `memory` 配置块：是否启用记忆、最大记忆条数、反思频率等 |
| `src/index.ts` 的 `onload()` | 初始化记忆目录和核心档案（如果不存在则创建默认模板） |

#### 5.1.6 第一阶段交互流程

```
用户发送消息
  → ai-chat.ts 拦截
    → retriever.retrieveRelevantMemories(query)
      → 关键词匹配（episodic JSONL 文件遍历）
      → 返回 Top-5 记忆
    → injector.injectMemories(messages, memories)
      → 在 system prompt 后附加记忆段落
    → 调用 opencode-provider.ts 正常发送
  → 用户收到回复，继续对话
  → 用户关闭会话（或切换会话）
    → ai-sidebar.svelte 调用 saveCurrentSession()
    → extractor.extractEpisodicMemory(sessionMessages)
      → 调用 OpenCode API 生成摘要
      → 在思源中创建新文档 `{rootPath}/情景记忆/2026/05/27-主题.md`
```

#### 5.1.7 第一阶段的限制（可接受）

- 检索依赖思源全文搜索和关键词匹配（无向量语义检索）
- 核心档案是静态的，不会自动更新（需要用户手动编辑，或等第二阶段）
- 检索性能依赖思源笔记的搜索索引

---

### 第二阶段：主动反思（让 Agent "会进化"）

目标：Agent 能定期回顾历史，自动更新核心档案，表现出"成长感"。

#### 5.2.1 反思触发器

- **计数触发**：每完成 N 次对话（默认 10 次），触发一次反思
- **定时触发**：每天首次打开插件时，检查是否需要反思
- **手动触发**：用户在设置面板点击"立即反思"

#### 5.2.2 反思流程（`src/memory/reflection.ts`）

1. **收集素材**：读取最近一段时间（如最近 10 次）的情景记忆条目
2. **生成反思 prompt**：
   ```
   你是一位善于观察和总结的心理学家。以下是 Agent 最近和用户的对话摘要：
   [episodic memories...]

   当前 Agent 的核心档案：
   [core profile...]

   请分析：
   1. 用户的行为模式或偏好是否有变化？
   2. 是否有新的重要信息需要加入核心档案？
   3. 核心档案中是否有过时的信息需要更新或删除？
   4. Agent 的自我认知是否需要调整？

   请返回更新后的核心档案 JSON，以及你做出这些修改的理由。
   ```
3. **调用模型**：将 prompt 发送给 OpenCode
4. **应用更新**：解析返回的 Markdown/JSON，更新思源笔记中的核心档案文档（`appendBlock` 或 `updateBlock`）
5. **记录日志**：在核心档案的"成长记录"章节追加本次反思记录

#### 5.2.3 记忆压缩（Memory Compaction）

随着时间推移，情景记忆会无限增长。需要定期压缩：

- **月级压缩**：每月月底，将该月的多篇情景记忆文档合并为 1 篇"月度摘要"文档
- **遗忘机制**：重要性评分低于阈值（如 0.3）且时间超过 3 个月的记忆，可归档到 `{rootPath}/情景记忆/归档/` 或删除
- **压缩流程**：
  ```
  读取 /AI记忆/情景记忆/2026/05/ 下的所有记忆文档（50 篇）
    → 调用模型："请将这些对话摘要合并为 5 条月度大事记"
    → 创建新文档 /AI记忆/情景记忆/2026/05-月度摘要.md
    → 将原文档移动到 /AI记忆/情景记忆/归档/2026/05/
  ```

#### 5.2.4 与现有 soulDocId 的融合

现有代码中已有一个 `soulDocId` 字段（`defaultSettings.ts`），它指向一个思源文档，作为 Agent 的"灵魂"。新设计直接将其升级：

- **soulDocId 即核心档案**：如果用户已设置 `soulDocId`，优先使用它作为核心档案文档
- **双向链接**：核心档案文档中可以嵌入查询，动态引用相关的情景记忆文档（利用思源的嵌入块功能）
- **用户完全掌控**：因为记忆就是普通笔记，用户可以用思源的所有功能（标签、属性、模板、导出等）来管理 Agent 的记忆

## 6. 数据安全与隐私

- **笔记即记忆**：所有记忆数据就是思源笔记文档，完全本地存储，随思源同步策略走
- **模型侧隐私**：记忆提取和反思需要调用 OpenCode API，传输的是对话摘要而非完整原始对话（用户可配置是否启用）
- **敏感信息过滤**：在提取记忆前，增加简单规则过滤密码、API Key 等敏感信息（正则匹配 + 模型判断）
- **用户可控**：
  - 设置中可随时关闭记忆功能
  - 可一键清除所有记忆（删除 `rootPath` 下的所有文档）
  - 可直接在思源笔记中查看、编辑、删除任何记忆文档

## 7. UI/UX 设计（简述）

### 7.1 设置面板扩展

在现有设置面板中新增"记忆"标签页：

- **启用长期记忆**：开关
- **自动提取记忆**：每次会话结束后自动提取（关则需手动点击"记住这次对话"）
- **反思频率**：每 N 次对话 / 每天 / 手动
- **记忆数量上限**：情景记忆最大条数（超出后触发压缩）
- **记忆存储路径**：选择/输入思源笔记中的目录路径（如 `/AI记忆/`）
- **查看核心档案**：点击后在思源中打开核心档案文档
- **查看情景记忆**：点击后在思源中打开情景记忆目录
- **清除所有记忆**：危险操作，需二次确认

### 7.2 聊天界面提示

- 当 Agent 使用了记忆时，在回复下方显示一个小标签："基于 X 条相关记忆"
- 用户 hover 标签时，弹出检索到的记忆摘要列表（增加透明度和信任感）

## 8. 风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| 记忆提取的 API 调用增加费用/延迟 | 中 | 提取过程异步进行，不阻塞用户；用户可关闭自动提取 |
| 核心档案被模型错误更新 | 高 | 第二阶段才启用自动更新；更新前做 JSON Schema 校验；保留历史版本 |
| 记忆注入导致 prompt 过长 | 中 | 严格限制检索条数（Top-5）和单条长度；注入前计算 token 数，超长则摘要 |
| 关键词检索精度有限 | 中 | 依赖用户清晰表达 + 思源全文搜索；后续可考虑轻量方案 |
| 用户担心隐私 | 中 | 本地存储 + 敏感信息过滤 + 用户完全可控的开关和清除 |
| 多设备同步问题 | 低 | 依赖 SiYuan 自带的同步机制（如果用户开启），记忆文件随数据目录同步 |

## 9. 与现有架构的兼容性

- **不破坏现有会话机制**：记忆层是可选的增强模块，关闭后插件行为与现在完全一致
- **不修改 OpenCode provider**：记忆注入发生在 `ai-chat.ts` 层，provider 仍然只负责发送和接收消息
- **复用现有存储机制**：使用思源 API（`createDocWithMd`、`appendBlock` 等）存储记忆文档
- **复用 soulDocId**：不新增概念，将已有的"灵魂文档"升级为记忆的核心档案载体

## 10. 成功标准

**第一阶段完成后**：
- [ ] 用户连续两天和 Agent 对话，第二天 Agent 能提到第一天聊过的内容
- [ ] 每次对话结束后，思源笔记的 `记忆存储路径/情景记忆/` 下自动生成一篇新文档
- [ ] 关闭记忆开关后，插件行为与 v0.0.2 完全一致

**第二阶段完成后**：
- [ ] Agent 在对话中表现出对用户偏好的适应（例如"我知道你不喜欢长解释，我直接给代码"）
- [ ] 核心档案文档能随反思自动更新，用户可在思源中查看变更历史
- [ ] 用户可以直接在思源笔记中编辑核心档案，Agent 会读取用户的修改
